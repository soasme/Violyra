## Feature 1: Prompt Architecture Refactor (#5)

### Goal

Eliminate duplicated multi-shot and single-shot prompt logic from model-specific skills by extracting it into two shared, model-agnostic skills. Model-specific skills become thin wrappers that supply model-specific config.

### New Skills

**`writing-multishot-prompt`**

Accepts a shot list and a `model_config` block. Produces a chained prompt string using the model's native multi-shot delimiter.

`model_config` fields:
- `delimiter` â€” string inserted between shots (e.g. `"Cut to:"` for Veo 3.1, `"Shot switch."` for Seedance 2.0)
- `max_words_per_shot` â€” word budget per shot segment
- `native_audio` â€” boolean; whether to include audio cue instructions

**`writing-singleshot-prompt`**

Single-shot fallback for models without native multi-shot support. Accepts one shot description and model style hints. Produces one optimized prompt string. Called iteratively by the orchestrator, once per shot.

### Updated Skills

| Skill | Change |
|---|---|
| `writing-veo31-prompt` | Strips multi-shot section; delegates to `writing-multishot-prompt` with `delimiter: "Cut to:"`, `native_audio: true`. Retains Veo-specific rules: cinematic language, physical realism, R2V guidelines, audio cue formats. |
| `writing-seedance20-prompt` | Strips multi-shot section; delegates to `writing-multishot-prompt` with `delimiter: "Shot switch."`, `native_audio: false`. Retains Seedance-specific rules: motion-first, degree adverbs, intensity modifiers. |
| `writing-seedance15-prompt` | Delegates to `writing-singleshot-prompt`. Retains Seedance 1.5-specific constraints. |

### Scope

SKILL.md changes only. No script changes.

### Acceptance Criteria

- [ ] `writing-multishot-prompt` skill exists and is model-agnostic
- [ ] `writing-singleshot-prompt` skill exists as a fallback
- [ ] Veo 3.1 and Seedance 2.0 skills delegate to `writing-multishot-prompt`
- [ ] Seedance 1.5 skill delegates to `writing-singleshot-prompt`
- [ ] No duplicate multi-shot prompt logic remains in model-specific skills

## Feature 2: External Shared Asset Directory (#6)

### Goal

Allow projects to reference assets from a shared library without duplicating files. Resolution is transparent to all skills.

### Config Change

`project.json` gains an optional `sharedAssetDirs` field (array of absolute paths):

```json
{
  "$schemaVersion": "1.0",
  "title": "My Video",
  "sharedAssetDirs": ["/Users/me/studio-assets", "/Volumes/NAS/shared"]
}
```

Existing projects without this field are unaffected.

### Script Change

**`skills/lib/pack-utils.js`** â€” new export:

```js
resolveAsset(projectDir, relativePath)
```

Reads `{projectDir}/project.json` internally to obtain `sharedAssetDirs` (defaults to `[]` if field absent).

Resolution order:
1. `{projectDir}/{relativePath}` â€” project-local (current behavior)
2. `{sharedAssetDir}/{relativePath}` for each dir in `sharedAssetDirs`, in config order
3. Throws if not found in any location

All existing pack-loading calls that currently construct paths with `join(projectDir, relativePath)` switch to `resolveAsset(...)`.

### Skill Change

**`setup-video-project/SKILL.md`** â€” documents the optional `sharedAssetDirs` field and its resolution semantics.

### Acceptance Criteria

- [ ] `resolveAsset` helper added to `pack-utils.js`
- [ ] Asset resolution checks project-local first, then shared dirs in order
- [ ] `setup-video-project` documents the new optional config field
- [ ] Existing behavior unchanged when `sharedAssetDirs` is absent
- [ ] Tests cover: local hit, shared dir fallback, not-found error

## Feature 3: `using-wangp-model` Skill (#7)

### Goal

New skill wrapping the `wangp` CLI for offline batch video and audio inference. Follows the shape of `using-replicate-model` and `using-falai-model`.

### Skill: `using-wangp-model/SKILL.md`

Sections:

**Authentication**
- No remote API token required
- Documents any required env vars (e.g. `WANGP_MODEL_DIR` for local model weights)

**Workflow**
1. Select model name and locate input assets
2. Construct `wangp` command with input/output flags
3. Submit batch job; capture job ID from stdout
4. Poll `wangp status <job-id>` until `completed` or `failed`
5. Confirm output files exist; return output paths

**Command Patterns**
- Video inference section (flags, example commands)
- Audio inference section (flags, example commands)
- Output directory is always caller-supplied

**Guidelines**
- Validate input asset paths exist before submitting
- Output files are local â€” no download step needed (unlike Replicate/FAL)
- On failure: log `failed` event with `wangp` error output in `notes`

**Logging**
- `inputs`: `model`, `input_paths` (array), `output_dir`
- `outputs`: `output_paths` (array)
- `notes`: error message on failure

### Acceptance Criteria

- [ ] `using-wangp-model` skill created
- [ ] Covers video inference and audio inference use cases
- [ ] Example commands included for both modes
- [ ] Logging section present and consistent with logging-guide.md

## Feature 4: Inference Caching via Manifest (#8)

### Goal

Before calling any inference API or CLI, model skills check a project-local `manifest.json`. Cache hits skip inference and return the existing output path directly.

### New Library: `skills/lib/manifest-utils.js`

Exports:

| Function | Signature | Description |
|---|---|---|
| `computeCacheKey` | `(prompt, modelId, params) â†’ string` | SHA-256 of `JSON.stringify({prompt, modelId, params})` with stable key sort. Returns `"sha256:{hex}"`. |
| `readManifest` | `(projectDir) â†’ manifest` | Reads `{projectDir}/manifest.json`; returns empty manifest if absent. |
| `writeManifest` | `(projectDir, manifest) â†’ void` | Writes `{projectDir}/manifest.json`. |
| `findEntry` | `(manifest, cacheKey) â†’ entry \| null` | Returns matching entry or `null`. |
| `isEntryValid` | `(entry, projectDir) â†’ boolean` | Returns `true` if `{projectDir}/{entry.output}` exists on disk. |
| `addEntry` | `(manifest, entry) â†’ manifest` | Appends entry; returns updated manifest (does not write). |

### Manifest Format

`{project_dir}/manifest.json` â€” JSON, append-only in practice:

```json
{
  "$schemaVersion": "1.0",
  "entries": [
    {
      "cache_key": "sha256:abc123...",
      "prompt": "Wide establishing shot of a rain-slicked street...",
      "model": "bytedance/seedance-2.0-pro",
      "params_hash": "sha256:def456...",
      "output": "chapters/ch1/shot_01.mp4",
      "created_at": "2026-03-29T10:00:00Z"
    }
  ]
}
```

`params_hash` is a separate SHA-256 of the inference params object (resolution, duration, seed, etc.), kept separate from `cache_key` for human readability.

### Model Skill Updates

Each of `using-replicate-model`, `using-falai-model`, and `using-wangp-model` gains a **Caching** section in their SKILL.md:

**Lookup flow (before calling inference):**
1. Compute cache key from prompt + model ID + inference params
2. Call `readManifest(projectDir)` and `findEntry(manifest, cacheKey)`
3. If entry found and `isEntryValid(entry, projectDir)` â†’ return `entry.output`; log `completed` with `notes: "cache_hit"`
4. If no entry or stale â†’ proceed with inference

**Write flow (after successful inference):**
1. Call `addEntry` with new entry (cache key, prompt, model, params hash, output path, timestamp)
2. Call `writeManifest` to persist

### Cache Invalidation

- **Delete output file** â†’ `isEntryValid` returns false â†’ treated as cache miss on next run
- **Delete manifest entry** â†’ manual edit of `manifest.json`; next run re-infers
- No TTL, no max-entry limit â€” manual invalidation only

### Tests

`skills/lib/manifest-utils.__test__.js` covers:
- `computeCacheKey` is deterministic regardless of params key order
- `findEntry` returns correct entry on hit, `null` on miss
- `isEntryValid` returns `false` when output file is absent
- `readManifest` returns empty manifest when file does not exist
- `addEntry` + `writeManifest` + `readManifest` round-trip

### Acceptance Criteria

- [ ] `manifest-utils.js` implemented with all exports above
- [ ] `manifest-utils.__test__.js` passes (all cases above)
- [ ] `using-replicate-model`, `using-falai-model`, `using-wangp-model` each have a Caching section
- [ ] Cache hit skips inference and logs `cache_hit` in notes
- [ ] Stale entries (output file deleted) are treated as cache misses
- [ ] `manifest.json` uses `$schemaVersion: "1.0"` consistent with pack files
