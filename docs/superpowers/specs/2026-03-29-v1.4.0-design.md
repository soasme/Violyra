## Feature 1: Prompt Architecture Refactor (#5)

### Goal

Replace three model-specific prompt skills with one unified `writing-video-prompt` skill. Model-specific guidance moves into `references/<model>.md` files that the skill directs the agent to consult.

### Deleted Skills

- `writing-veo31-prompt`
- `writing-seedance20-prompt`
- `writing-seedance15-prompt`

### New Skill: `writing-video-prompt`

A single model-agnostic prompt-writing skill. It:

1. Accepts a shot list (or single shot description) and a target model name
2. Instructs the agent to consult `references/<model>.md` for model-specific rules (delimiter, word budget, audio support, motion language, etc.)
3. Applies universal principles: cinematic language, sequential action, subject clarity, physical realism where applicable
4. Produces a prompt string ready for the inference skill — either a chained multi-shot prompt (if the model supports it) or a single-shot prompt (if not), as determined by the model reference

### New Reference Files

Model-specific guidance extracted from the deleted skills:

| File | Content |
|---|---|
| `references/veo-3.1-prompt.md` | Multi-shot delimiter (`Cut to:`), word budget, native audio cue format, R2V rules, physical realism, cinematic language vocabulary |
| `references/seedance-2.0-prompt.md` | Multi-shot delimiter (`Shot switch.`), motion-first principle, degree adverbs, intensity modifiers |
| `references/seedance-1.5-prompt.md` | Single-shot only, Seedance 1.5 constraints |

Reference files live alongside the existing `references/` entries (e.g. `seedance-1.5-pro.md`, `veo-3.1.md`).

### Scope

SKILL.md additions/deletions and new `references/` markdown files. No script changes.

### Acceptance Criteria

- [ ] `writing-video-prompt` skill created
- [ ] `writing-veo31-prompt`, `writing-seedance20-prompt`, `writing-seedance15-prompt` deleted
- [ ] `references/veo-3.1-prompt.md`, `references/seedance-2.0-prompt.md`, `references/seedance-1.5-prompt.md` created
- [ ] `writing-video-prompt` instructs agent to consult the appropriate reference file by model name
- [ ] No model-specific prompt logic remains in the skill itself

## Feature 2: External Shared Asset Directory (#6)

### Goal

Allow projects to reference assets from a shared library without duplicating files. Resolution is transparent to all skills.

### Config Change

`project.json` gains an optional `assetDirs` field (array of paths, absolute or relative to `project.json`):

```json
{
  "$schemaVersion": "1.0",
  "title": "My Video",
  "assetDirs": [".", "/Users/me/studio-assets", "/Volumes/NAS/shared"]
}
```

Default when field is absent: `["."]` — same directory as `project.json`, preserving existing behavior exactly.

### Script Change

**`skills/lib/pack-utils.js`** — new export:

```js
resolveAsset(projectDir, relativePath)
```

Reads `{projectDir}/project.json` internally to obtain `assetDirs` (defaults to `["."]` if field absent). Resolves each entry relative to `projectDir` if not absolute.

Resolution order:
1. Each dir in `assetDirs`, in config order — first match wins
2. Throws if not found in any location

**Internal vs external dirs:**

A dir is *internal* if it is `projectDir` itself or a subdirectory of it. All other dirs are *external*.

- Agents **read** from any dir (internal or external).
- Agents **write** to internal dirs by default. Output files, generated assets, and updated packs land inside `projectDir` unless the user explicitly instructs the agent to move or copy assets across dirs (e.g. promoting a generated asset into a shared library).
- `resolveAsset` is for reads only. Write paths are constructed with `join(projectDir, ...)` directly in normal production flow.

All existing pack-loading calls that currently construct paths with `join(projectDir, relativePath)` switch to `resolveAsset(...)`.

### Skill Change

**`setup-video-project/SKILL.md`** — documents the optional `sharedAssetDirs` field and its resolution semantics.

### Acceptance Criteria

- [ ] `resolveAsset` helper added to `pack-utils.js`
- [ ] Asset resolution searches `assetDirs` in order, first match wins
- [ ] `setup-video-project` documents the optional `assetDirs` field
- [ ] Default `["."]` preserves existing behavior when field is absent
- [ ] Tests cover: default single-dir hit, multi-dir fallback, not-found error

## Feature 3: `using-wangp-model` Skill (#7)

### Goal

New skill wrapping the `wangp` CLI for offline batch video and audio inference. Follows the shape of `using-replicate-model` and `using-falai-model`.

### Skill: `using-wangp-model/SKILL.md`

Sections:

**Authentication**
- No remote API token required
- Documents any required env vars (e.g. `WANGP_MODEL_DIR` for local model weights)

**Workflow**
1. Select model name and locate input assets
2. Construct `wangp` command with input/output flags
3. Submit batch job; capture job ID from stdout
4. Poll `wangp status <job-id>` until `completed` or `failed`
5. Confirm output files exist; return output paths

**Command Patterns**
- Video inference section (flags, example commands)
- Audio inference section (flags, example commands)
- Output directory is always caller-supplied

**Guidelines**
- Validate input asset paths exist before submitting
- Output files are local — no download step needed (unlike Replicate/FAL)
- On failure: log `failed` event with `wangp` error output in `notes`

**Logging**
- `inputs`: `model`, `input_paths` (array), `output_dir`
- `outputs`: `output_paths` (array)
- `notes`: error message on failure

### Acceptance Criteria

- [ ] `using-wangp-model` skill created
- [ ] Covers video inference and audio inference use cases
- [ ] Example commands included for both modes
- [ ] Logging section present and consistent with logging-guide.md

## Feature 4: Inference Caching via Manifest (#8)

### Goal

Before calling any inference API or CLI, model skills check a project-local `manifest.json`. Cache hits skip inference and return the existing output path directly.

### New Library: `skills/lib/manifest-utils.js`

Exports:

| Function | Signature | Description |
|---|---|---|
| `computeCacheKey` | `(prompt, modelId, params) → string` | SHA-256 of `JSON.stringify({prompt, modelId, params})` with stable key sort. Returns `"sha256:{hex}"`. |
| `readManifest` | `(projectDir) → manifest` | Reads `{projectDir}/manifest.json`; returns empty manifest if absent. |
| `writeManifest` | `(projectDir, manifest) → void` | Writes `{projectDir}/manifest.json`. |
| `findEntry` | `(manifest, cacheKey) → entry \| null` | Returns matching entry or `null`. |
| `isEntryValid` | `(entry, projectDir) → boolean` | Returns `true` if `{projectDir}/{entry.output}` exists on disk. |
| `addEntry` | `(manifest, entry) → manifest` | Appends entry; returns updated manifest (does not write). |

### Manifest Format

`{project_dir}/manifest.json` — JSON, append-only in practice:

```json
{
  "$schemaVersion": "1.0",
  "entries": [
    {
      "cache_key": "sha256:abc123...",
      "prompt": "Wide establishing shot of a rain-slicked street...",
      "model": "bytedance/seedance-2.0-pro",
      "params_hash": "sha256:def456...",
      "output": "chapters/ch1/shot_01.mp4",
      "created_at": "2026-03-29T10:00:00Z"
    }
  ]
}
```

`params_hash` is a separate SHA-256 of the inference params object (resolution, duration, seed, etc.), kept separate from `cache_key` for human readability.

### Model Skill Updates

Each of `using-replicate-model`, `using-falai-model`, and `using-wangp-model` gains a **Caching** section in their SKILL.md:

**Lookup flow (before calling inference):**
1. Compute cache key from prompt + model ID + inference params
2. Call `readManifest(projectDir)` and `findEntry(manifest, cacheKey)`
3. If entry found and `isEntryValid(entry, projectDir)` → return `entry.output`; log `completed` with `notes: "cache_hit"`
4. If no entry or stale → proceed with inference

**Write flow (after successful inference):**
1. Call `addEntry` with new entry (cache key, prompt, model, params hash, output path, timestamp)
2. Call `writeManifest` to persist

### Cache Invalidation

- **Delete output file** → `isEntryValid` returns false → treated as cache miss on next run
- **Delete manifest entry** → manual edit of `manifest.json`; next run re-infers
- No TTL, no max-entry limit — manual invalidation only

### Tests

`skills/lib/manifest-utils.__test__.js` covers:
- `computeCacheKey` is deterministic regardless of params key order
- `findEntry` returns correct entry on hit, `null` on miss
- `isEntryValid` returns `false` when output file is absent
- `readManifest` returns empty manifest when file does not exist
- `addEntry` + `writeManifest` + `readManifest` round-trip

### Acceptance Criteria

- [ ] `manifest-utils.js` implemented with all exports above
- [ ] `manifest-utils.__test__.js` passes (all cases above)
- [ ] `using-replicate-model`, `using-falai-model`, `using-wangp-model` each have a Caching section
- [ ] Cache hit skips inference and logs `cache_hit` in notes
- [ ] Stale entries (output file deleted) are treated as cache misses
- [ ] `manifest.json` uses `$schemaVersion: "1.0"` consistent with pack files
