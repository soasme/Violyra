# v1.3.0 Design Spec

Date: 2026-03-29

## Overview

v1.3.0 introduces three infrastructure pillars that improve the reliability of agents working on violyra and the observability and quality measurement of produced videos:

1. **Harness Engineering** — repo-level enforcement mechanisms and golden principles for agents working on the codebase
2. **Observability** — JSONL logging woven into every skill so pipeline runs are traceable
3. **Performance Scoring** — a new `scoring-narrative-quality` skill that scores assembled videos against a 5-dimension narrative rubric

No new user-facing pipeline skills are added. All changes are either infra-layer (harness, observability) or a single new scoring skill.

---

## Pillar 1 — Harness Engineering

### Purpose

Make AI agents working *on* violyra more reliable through deterministic enforcement of golden principles. Invisible to skill consumers.

### Components

**`AGENTS.md`** (repo root)

Authoritative reference for any coding agent working on the repo. Encodes:
- Skill naming convention: `gerund-object` (e.g., `writing-seedance20-prompt`)
- Skill `description` field: max 200 characters
- Every script requires a `__test__.js` companion in the same directory
- Keep logic in reusable functions; side effects only in `main()`
- No LLM calls in code
- `pnpm test` must pass before any commit to `skills/`
- All skill changes must pass `pnpm lint-skills` before commit

**`scripts/lint-skills.js`**

Deterministic linter for skill file integrity. Checks every `skills/*/SKILL.md`:
- Has valid YAML frontmatter with `name` and `description` fields
- `description` is ≤ 200 characters
- Skill directory name matches the `name` field in frontmatter
- Skill name follows kebab-case slug matching gerund-object convention (regex: `/^[a-z]+-[a-z0-9]+(-[a-z0-9]+)*$/`; digits allowed in non-first segments to accommodate names like `writing-seedance15-prompt`)

Reports violations to stdout; exits non-zero on any failure.

**`scripts/lint-skills.__test__.js`**

Vitest tests for the linter itself, covering valid and invalid cases for each check.

**`.claude/settings.json` hooks**

Pre-tool hook that runs `pnpm lint-skills` before any file write targeting `skills/`. Blocks malformed skill additions automatically.

### New files

| Action | Path |
|---|---|
| Create | `AGENTS.md` |
| Create | `scripts/lint-skills.js` |
| Create | `scripts/lint-skills.__test__.js` |
| Update | `.claude/settings.json` (add hook) |
| Update | `package.json` (add `lint-skills` script) |

## Pillar 2 — Observability (JSONL logging)

### Purpose

Make every skill invocation traceable. The coding agent writes a JSONL entry when it starts a skill and when it completes, creating an append-only audit trail per production project.

### Log file location

```
{project_dir}/logs/production.jsonl
```

One file per production project. Append-only. Never truncated or rewritten.

### Log entry schema

Each line is a JSON object with the following fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `ts` | ISO 8601 string | yes | Timestamp of the event |
| `skill` | string | yes | Skill name (matches `name` in frontmatter) |
| `event` | string | yes | `invoked`, `completed`, `failed`, `retried`, or `scored` |
| `reason` | string | yes (on `invoked`) | Why the agent called this skill — free text written by the agent |
| `inputs` | object | no | Key input parameters relevant to the skill invocation |
| `outputs` | object | no | Key output artifacts or results |
| `notes` | string | no | Agent observations, decisions, or anomalies |

**Example entries:**

```jsonl
{"ts":"2026-03-29T12:00:00Z","skill":"writing-veo31-prompt","event":"invoked","reason":"Shot 3 needs a close-up prompt for the bridge section","inputs":{"shot_id":"ch1-s03","mode":"text_to_video"}}
{"ts":"2026-03-29T12:01:14Z","skill":"writing-veo31-prompt","event":"completed","outputs":{"prompt_length":142,"shot_id":"ch1-s03"},"notes":"Used R2V mode due to reference image availability"}
{"ts":"2026-03-29T12:05:00Z","skill":"using-falai-model","event":"invoked","reason":"Generating video for shot ch1-s03 using veo3.1","inputs":{"model":"fal-ai/veo3.1/text-to-video","shot_id":"ch1-s03"}}
{"ts":"2026-03-29T12:07:33Z","skill":"using-falai-model","event":"failed","notes":"Timeout after 120s. Retrying."}
{"ts":"2026-03-29T12:07:35Z","skill":"using-falai-model","event":"retried","inputs":{"model":"fal-ai/veo3.1/text-to-video","shot_id":"ch1-s03"}}
{"ts":"2026-03-29T12:10:01Z","skill":"using-falai-model","event":"completed","outputs":{"output_url":"https://...","duration_s":8,"shot_id":"ch1-s03"}}
```

### Shared logging reference

**`skills/lib/logging-guide.md`** — defines the schema and examples in one place. Each skill's `SKILL.md` adds a brief **Logging** section referencing this guide and specifying the key `inputs` and `outputs` fields relevant to that skill.

### Scope of change

Every existing `SKILL.md` (30+ files) gets a **## Logging** section added as the last section of the file (after all skill content). The section is 3–5 lines: what `inputs` and `outputs` fields to include for this skill, and a reference to `skills/lib/logging-guide.md` for the full schema. The coding agent appends an `invoked` entry before executing the skill and a `completed`/`failed` entry when done.

### New / modified files

| Action | Path |
|---|---|
| Create | `skills/lib/logging-guide.md` |
| Update | All 30+ `skills/*/SKILL.md` files (add Logging section) |

## Pillar 3 — Performance Scoring

### Purpose

Evaluate a compiled video's narrative and retention quality using a 5-dimension rubric. Runs after `compiling-video`. Complements `retention-driven-development` (per-shot simulation) by scoring the assembled whole.

### New skill: `scoring-narrative-quality`

**Trigger:** After `compiling-video` produces the final output file.

**Inputs:** Path to the compiled video, path to the project's `production.jsonl` log (for context), and the original storyboard/shot list.

**Process:**

1. Read `production.jsonl` to understand what was generated and any retries or failures
2. Review the compiled video against the storyboard
3. Score each of the 5 dimensions (0–20 each)
4. Write a single `scored` JSONL entry to `production.jsonl`
5. If composite score < 70, recommend which `retention-driven-development` pass to run next (based on the lowest-scoring dimension)

### 5-dimension rubric

| Dimension | What it measures | Max |
|---|---|---|
| `hook` | Does the opening 3s compel continued watching? | 20 |
| `pacing` | Do cuts and shot durations match the emotional rhythm? | 20 |
| `emotional_arc` | Is there a clear build, peak, and resolution? | 20 |
| `visual_variety` | Do shot types, angles, and framing change enough to sustain attention? | 20 |
| `payoff` | Does the ending feel earned relative to the setup? | 20 |

**Composite: sum of all five dimensions (0–100).**

### Score JSONL entry

```jsonl
{"ts":"2026-03-29T14:22:00Z","skill":"scoring-narrative-quality","event":"scored","scores":{"hook":16,"pacing":14,"emotional_arc":18,"visual_variety":12,"payoff":15},"composite":75,"notes":"Visual variety dragged by 4 consecutive medium shots in the chorus"}
```

### Relationship to `retention-driven-development`

| Skill | Scope | When to run |
|---|---|---|
| `retention-driven-development` | Per-shot viewer simulation; replaces weak shots | During production, before `compiling-video` |
| `scoring-narrative-quality` | Whole-video narrative evaluation; produces a composite score | After `compiling-video` |

If `scoring-narrative-quality` composite < 70, it identifies the weakest dimension and recommends a targeted `retention-driven-development` pass before recompiling.

## What is NOT in v1.3.0

- No new video generation model integrations
- No new pipeline orchestration skills
- No UI or dashboard for the JSONL logs
- No automated scoring scripts (scoring is agent-driven, not code-driven)
