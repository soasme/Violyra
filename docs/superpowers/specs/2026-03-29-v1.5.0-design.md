# v1.5.0 Design Spec

**Date:** 2026-03-29
**Branch:** codex/feat-v1.5.0
**Source Feedback:** `/Users/soasme/github.com/soasme/violyra/assets/issues.md`

---

## Overview

v1.5.0 rebuilds Violyra's idea-to-execution workflow around the same approval-first behavior that makes `superpowers` useful, but adapts it for video production instead of software implementation.

The problem is not that the current pipeline cannot produce a video. It can. The problem is that the user has to infer too much:

- what happens first
- what happens next
- which artifact is for the user versus for automation
- when reference-image work begins
- when retention and review loops happen
- why `executing-video-plan` is blocked

The chosen design is a dual-artifact workflow:

1. a collaborative, approval-gated video idea spec
2. a human-readable production plan plus a machine-readable execution manifest

`executing-video-plan`, `docs/usage.md`, and the plan-writing skill all read from the same workflow model so the repo stops drifting between documentation, planning, and execution behavior.

---

## Alternatives Considered

### Option A: Light documentation patch only

Add `docs/usage.md`, add a short README section, and leave the skills mostly as they are.

**Pros**
- Fastest to ship
- Lowest code churn

**Cons**
- Does not fix the core mismatch between `writing-video-plan` and `executing-video-plan`
- Keeps key behavior implicit in agent judgment
- Does not import the strongest `superpowers` behavior: approval gates and artifact review loops

### Option B: Superpowers-style planning with a video-specific execution manifest

Keep the current Violyra workflow shape, but make it explicit and shared:

- `brainstorming-video-idea` becomes negotiation-first
- `writing-video-plan` emits both a markdown runbook and a machine-readable phase manifest
- `executing-video-plan` becomes state-aware and phase-driven

**Pros**
- Fixes the user-facing workflow confusion
- Preserves current Violyra concepts and asset layout
- Gives both humans and agents the artifact they need

**Cons**
- Requires touching several related skills together
- Needs one shared workflow definition to avoid drift

### Option C: Full workflow engine

Replace the current skill flow with one top-level orchestrator that owns every step end-to-end.

**Pros**
- Potentially the cleanest long-term automation story

**Cons**
- Too large for this iteration
- Would bury useful existing skills under a new abstraction layer
- Higher risk of breaking a pipeline that already works

### Recommended Option

Choose **Option B**.

It solves the concrete failures from the first end-to-end run without replacing the whole system. It also matches the user's request most closely: keep the `superpowers` quality bar for brainstorming and planning, but adapt it to video production.

## Pillar 1: `brainstorming-video-idea` Becomes Approval-First

### Goal

Make `brainstorming-video-idea` behave like `superpowers/brainstorming`: the agent should negotiate toward a better plan with the user before writing files or moving into project setup.

### Intended Behavior

`skills/brainstorming-video-idea/SKILL.md` is rebuilt around these rules:

1. Explore current context first.
   - Existing project workspace, existing assets, previous runs, target audience, platform assumptions.
2. Do not create project files or transition to setup before design approval.
3. Ask one clarifying question at a time.
4. Offer 2-3 creative directions with trade-offs before converging.
5. Present the design in sections scaled to the job:
   - audience and tone
   - characters and continuity requirements
   - music / lyrics assumptions
   - scene progression
   - production risks
6. After approval, write the agreed design doc and ask the user to review it before planning begins.

### Video-Specific Adaptation

The generic `superpowers` structure stays, but the section content changes for Violyra. The design conversation should optimize for video-production questions:

- what the viewer should feel in the first 3 seconds
- how recurring characters stay visually consistent
- whether the workflow needs reference images or start frames
- what scene count and cadence the song length implies
- where the likely regeneration risk lives

### Written Artifact

The approved design is written to the production workspace as:

`{project_dir}/docs/video-idea.md`

This remains the user-facing creative source of truth. It is not replaced by JSON.

### Non-Goals

- No automatic project setup during brainstorming
- No auto-transition into scene generation
- No attempt to finalize prompts during idea negotiation

### Acceptance Criteria

- [ ] `brainstorming-video-idea` has a hard gate against setup/planning before user approval
- [ ] The skill explicitly asks one question at a time
- [ ] The skill explicitly proposes 2-3 approaches with trade-offs
- [ ] The approved design is written to `{project_dir}/docs/video-idea.md`
- [ ] The skill asks the user to review the written idea doc before `writing-video-plan`

## Pillar 2: Video-Specific Spec Review And Visual Companion

### Goal

Bring the strongest supporting pieces from `superpowers/brainstorming` into Violyra so brainstorming is not just a better prompt, but a better workflow.

### `spec-document-reviewer-prompt.md`

Add a project-local reviewer prompt at:

`skills/brainstorming-video-idea/spec-document-reviewer-prompt.md`

It should review a video idea spec for:

- missing or contradictory audience / tone assumptions
- unclear character continuity rules
- scene ideas that do not match the song structure
- unbounded scope that would create multiple independent productions
- missing production-feasibility assumptions such as required source audio, reference-image needs, or model constraints

The approval bar should stay pragmatic: only block when the written spec would cause the next planning step to build the wrong thing.

### `visual-companion.md`

Add a project-local guide at:

`skills/brainstorming-video-idea/visual-companion.md`

It should mirror the `superpowers` browser companion workflow, but explain video-production use cases instead of UI mockups. The browser becomes useful for:

- comparing visual directions for a song
- showing character-sheet options
- comparing color palettes and art styles
- showing beat-to-scene strips
- comparing candidate reference frames
- previewing thumbnail directions

### Script Adaptation

Add a project-local `scripts/` set under `skills/brainstorming-video-idea/` derived from the `superpowers` helper server:

- `helper.js`
- `server.cjs`
- `start-server.sh`
- `stop-server.sh`
- `frame-template.html`

The server architecture stays the same. The adaptation is at the content contract and UI affordances:

- support image-board and frame-gallery layouts as first-class examples
- preserve click logging for option selection
- allow metadata on selected cards such as `scene_id`, `frame_id`, or `character_id`
- keep content-fragment authoring as the default

This is intentionally not a full design app. It stays lightweight and browser-local.

### Acceptance Criteria

- [ ] `brainstorming-video-idea` has a local `spec-document-reviewer-prompt.md`
- [ ] `brainstorming-video-idea` has a local `visual-companion.md`
- [ ] The local visual companion docs describe video-specific use cases
- [ ] The local scripts are adapted from `superpowers`, not referenced indirectly
- [ ] The companion remains optional and per-question, not mandatory for the whole session

## Pillar 3: `writing-video-plan` Becomes A Real Production Plan

### Goal

Rebuild `writing-video-plan` from the `superpowers/writing-plans` model so the user gets a real plan, not only a storyboard artifact.

### Core Output Contract

After an approved `video-idea.md`, `writing-video-plan` writes three artifacts:

1. `{project_dir}/assets/storyboard.json`
2. `{project_dir}/docs/video-plan.md`
3. `{project_dir}/assets/production-plan.json`

Each artifact has a different job:

- `storyboard.json` is the creative machine input
- `video-plan.md` is the human-readable runbook
- `production-plan.json` is the machine-readable execution contract

### Why Three Artifacts

The first run showed that one JSON storyboard is not enough for the user, but a markdown-only plan is not enough for automation. The repo needs both.

### `video-plan.md`

This file is the video-production analog to a `superpowers` implementation plan. It should be explicit enough that a user or agent can open it and know:

- what exists already
- what the next phase is
- which files each phase expects
- which files each phase produces
- what blocks a phase
- what skill normally runs that phase
- how to verify phase completion

The markdown plan should use phase-based checklists instead of software TDD tasks. Each phase must be concrete and free of placeholders.

Required phases in the default happy path:

1. project setup
2. source asset localization (`song.mp3`, lyrics, optional source notes)
3. production pipeline outputs (`chapter.json`, `shot-list.json`, `shot-details.json`, packs)
4. lyric alignment
5. reference-image and start-frame generation
6. video prompt writing
7. first-pass scene generation
8. draft compile
9. retention-driven-development pass
10. recompile
11. requesting-video-review
12. thumbnail generation
13. delivery

The plan may mark phases optional or skippable, but it must say when and why.

### `production-plan.json`

This file lets `executing-video-plan` operate on the real project state instead of guessing. It should be a small execution manifest, not a second storyboard.

Proposed structure:

```json
{
  "$schemaVersion": "1.0",
  "project_dir": "assets/coofy-farm-tour",
  "phases": [
    {
      "id": "source-assets",
      "title": "Localize source audio and lyrics",
      "default_skill": "aligning-lyrics",
      "requires": ["assets/lyrics.txt", "assets/song.mp3"],
      "produces": [
        "assets/aligned_lyrics.json",
        "assets/subtitle.srt",
        "assets/subtitle.lrc"
      ],
      "verification": [
        "aligned_lyrics.json exists",
        "alignment line count matches planned sung lines"
      ],
      "blocks": [
        "song.mp3 missing"
      ]
    }
  ]
}
```

The exact schema can be refined during implementation, but it must answer four questions deterministically:

1. what phase is this
2. what must exist before it can run
3. what should exist after it completes
4. what skill is normally responsible for it

### `plan-document-reviewer-prompt.md`

Add a project-local reviewer prompt at:

`skills/writing-video-plan/plan-document-reviewer-prompt.md`

It should adapt the `superpowers` plan review criteria for production planning:

- completeness of phases and artifacts
- alignment with `video-idea.md`
- phase ordering and dependency clarity
- buildability for an agent with zero project context
- explicit placement of reference-image, retention, and review stages

### Acceptance Criteria

- [ ] `writing-video-plan` writes `storyboard.json`, `video-plan.md`, and `production-plan.json`
- [ ] `video-plan.md` clearly states the default next step after planning
- [ ] `video-plan.md` includes reference-image/start-frame work as a first-class phase
- [ ] `video-plan.md` explicitly places retention and review after draft compile
- [ ] `writing-video-plan` has a local `plan-document-reviewer-prompt.md`
- [ ] No placeholder phases such as "generate assets later" or "review as needed"

## Pillar 4: `executing-video-plan` Becomes State-Aware

### Goal

Make the natural user move after planning actually work.

Today, `executing-video-plan` inspects the workspace too late and too narrowly, then reports blockers that are technically true but product-wise unhelpful. In v1.5.0 it should read the plan artifacts and continue from the next runnable phase.

### Intended Behavior

`skills/executing-video-plan/SKILL.md` is updated to consume:

- `{project_dir}/docs/video-plan.md`
- `{project_dir}/assets/production-plan.json`
- current workspace state

It then does one of three things:

1. **Run the next runnable phase** if all prerequisites exist.
2. **Route to the exact missing prerequisite** if a required user input is absent.
3. **Explain the blocked phase in plan terms** instead of generic filesystem complaints.

### Example Improvement

Instead of:

- `chapters is empty`
- `song.mp3 missing`

It should report:

- current phase: `source-assets`
- blocked because: `assets/song.mp3` is missing
- once provided, next skill: `aligning-lyrics`
- production pipeline phase remains pending and is not yet expected

Or, if `song.mp3` exists but `chapters/` is empty:

- current phase: `production-pipeline`
- next skill: `running-video-production-pipeline`
- expected outputs: `chapter.json`, `shot-list.json`, `shot-details.json`

### Relationship To `retention-driven-development`

`executing-video-plan` should treat retention and review as explicit late phases, not as informal advice after compile.

Default sequence after generation:

1. compile draft
2. run retention-driven-development
3. recompile
4. run requesting-video-review
5. generate thumbnail if delivery-ready

### Acceptance Criteria

- [ ] `executing-video-plan` reads `production-plan.json`
- [ ] After `writing-video-plan`, the executor can identify the next runnable phase without guessing
- [ ] Blocked messages name the current phase, missing prerequisite, and recommended next skill
- [ ] Retention and review are explicit phases in executor output
- [ ] The executor no longer treats a missing later-phase artifact as a hard error when the plan has not reached that phase

## Pillar 5: One Canonical Workflow Model Powers Docs, Planning, And Execution

### Goal

Stop the repo from maintaining three different truths:

- what the docs say
- what `writing-video-plan` writes
- what `executing-video-plan` expects

### Shared Model

Introduce one canonical workflow definition in repo code or data, consumed by:

- `docs/usage.md`
- `README.md` usage summary
- `writing-video-plan`
- `executing-video-plan`

The model should define:

- phase order and dependencies
- default skill per phase
- required versus optional artifacts
- skip rules
- verification rules

This does not have to be complex. A JSON or JS module is sufficient if it is deterministic and shared.

### Documentation Outputs

`docs/usage.md` becomes the full happy-path guide:

- prerequisites
- first command
- normal phase order
- what each skill produces
- when to branch into reference-image work
- when to run retention-driven-development
- when to run requesting-video-review
- when thumbnail generation happens

`README.md` gets a short `Usage` section that points to `docs/usage.md` and gives the quickstart in a few steps.

### Acceptance Criteria

- [ ] `docs/usage.md` exists and follows the canonical workflow model
- [ ] `README.md` has a short `Usage` section that points to `docs/usage.md`
- [ ] The documented happy path includes reference-image/start-frame generation
- [ ] The documented happy path includes retention and review in the correct order
- [ ] Planning and execution use the same phase names as the docs

## What Is Out Of Scope For v1.5.0

- changing generation model providers
- replacing the existing production pipeline skills
- logging overhaul beyond what the touched skills already require
- full automated QA and regeneration orchestration
- thumbnail model-quality improvements beyond documenting where thumbnail generation belongs

Those are real follow-up topics, but they are not required to solve the current workflow confusion.

---

## Delivery Order

1. Rebuild `brainstorming-video-idea` around the approval-first `superpowers` contract.
2. Add the local video-specific spec reviewer and visual companion assets.
3. Rebuild `writing-video-plan` so it emits both markdown and execution artifacts.
4. Add the shared workflow model.
5. Update `executing-video-plan` to consume the shared model and plan manifest.
6. Write `docs/usage.md` and the short README `Usage` section from the same model.

This order keeps the work incremental and prevents docs from being written against a workflow contract that does not exist yet.
